## Description

High parameter counts are a leading indicator of an SRP violation: the routine is juggling concerns that should be split. Long lists also bring positional confusion (the third Integer is which one again?) and force callers to fill defaults that may not apply. Refactor: group cohesive parameters into a record (TOrderInput), promote configuration to a builder pattern (FluentQuery.Where(...).OrderBy(...)), or split the method by axis (SaveOrder + SaveOrderItems). Threshold via [Detectors] LongParamMax / DetectorMaxParams (default 5). out/var parameters count just like value parameters; the implicit Self for class methods does not count.

## Examples

### Bad (triggers the rule)

```pascal
procedure SaveOrder(id, customer, address, city, zip, country, total, tax, shipping: ...);
```

### Good (idiomatic fix)

```pascal
procedure SaveOrder(const Order: TOrder);
```
