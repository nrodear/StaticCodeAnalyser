## v0.9.1 — SonarQube-Integration

Production-ready External-Issues-Push nach **SonarQube / SonarCloud**,
eine massive **Rule-Catalog-Expansion** (22 → 59 Regeln) und ein
**Severity-Single-Source-Refactor**. Detektor-Anzahl wächst von 41 auf
**59** mit 18 neuen Pascal- / Visibility- / SQL- / Format-Detektoren plus
allen 20 DFM-Detektoren, die jetzt endlich im dokumentierten Catalog
landen.

Drei Schwerpunkte:

1. **SonarQube-Integration** — Standalone-CLI (`--sonar-export`,
   `--sonar-test`, `--sonar-init`) plus IDE-Plugin
   (*Tools > Optionen > Sonar Integration*, *Send to Sonar* Kontext-
   Menüs) plus PowerShell-Helper-Scripts. DPAPI-verschlüsselter Token-
   Storage. MQR-Felder (`cleanCodeAttribute` + `impacts`) pro Regel.
2. **59 dokumentierte Regeln** — jeder emittierte `TFindingKind` hat
   jetzt einen vollständigen Catalog-Eintrag mit `cleanCodeAttribute`,
   `impacts`, Beispielen, Tags, CWE, OWASP. SARIF- und Sonar-Generic-
   Issue-Reports haben keine fehlenden Metadaten mehr.
3. **Severity Single-Source** — `TLeakFinding.SetKind(K)` zieht die
   Default-Severity aus `KIND_META`. 58 Detektor-Emit-Stellen vom
   alten Zwei-Zeilen-Pattern refaktoriert. Zwei stille Severity-
   Drift-Bugs als Nebeneffekt gefixt (`fkNilDeref` Error→Warning,
   `fkUnusedUses` Warning→Hint).

## 🛰 SonarQube-Integration (NEU)

End-to-End-Pipeline für den Push von SCA-Findings zu einem SonarQube-
Server.

### `uSonarConfig` — Vier-Quellen-Konfigurations-Resolver

Auflösungs-Reihenfolge (höchste Priorität zuerst), jede Quelle füllt
nur leere Felder:

| Quelle | Standalone | IDE-Plugin |
|---|---|---|
| CLI-Flags (`--sonar-host`, `--sonar-token`, ...) | ✅ | — |
| Environment (`SONAR_HOST_URL`, `SONAR_TOKEN`, ...) | ✅ | ✅ |
| `sonar-project.properties` im Projekt-Root | ✅ | ✅ |
| `%APPDATA%\StaticCodeAnalyser\analyser.ini` `[Sonar]` + `[SonarTokens]` | ✅ | ✅ |

Token-Storage auf Windows ist DPAPI-verschlüsselt (Current-User-
Scope) — nur derselbe Windows-User auf demselben Rechner kann
entschlüsseln. Non-Windows: Plaintext+Base64-Fallback mit `PT:`-
Marker, dokumentiert in [docs/sonar-config.md](../sonar-config.md).

### `--sonar-test` Health-Check

Mehrstufige Konnektivitäts-Probe:

```text
[OK]   DNS resolution: sonar.company.com -> 10.0.0.42
[OK]   HTTP /api/system/status: UP
[OK]   Token validation: valid
[OK]   Project access: my-delphi-project (visible)
Sonar connection healthy.
```

403-Disambiguation: wenn `/api/projects/search` mit 403 antwortet,
fragt ein Fallback-Request auf `/api/components/show` ob das Projekt
existiert (404) oder ob die Browse-Permission fehlt (403) — die
Fehlermeldung weist auf die richtige nächste Aktion hin.

### `--sonar-export` Generic-Issue-Format-Writer

```powershell
analyser.exe --path . --full --base-dir . --sonar-export sca-findings.json
```

Schreibt eine Sonar-Generic-Issue-Format-JSON mit allen Findings.
Jede Rule trägt:

- `engineId = "static-code-analyser"`
- `cleanCodeAttribute` (eines von `LAWFUL` / `LOGICAL` / `FOCUSED` / ...
  14-Wert-MQR-Taxonomie)
- `impacts: [{softwareQuality, severity}]` (MQR-Modus)
- `effortMinutes` abgeleitet aus FindingType

Fallback-Pfad: falls der Catalog nicht geladen wurde (selten, z.B.
korrupte `rules/sca-rules.json`), emittiert der Exporter das Legacy-
`type`-Feld aus `KIND_META.FindingType` stattdessen — Sonar
akzeptiert die JSON trotzdem.

### `--sonar-init` Projekt-Template

```powershell
analyser.exe --sonar-init --path .
```

Schreibt ein `sonar-project.properties`-Template mit sinnvollen
Defaults und der Zeile
`sonar.externalIssuesReportPaths=sca-findings.json` ready zur
Verwendung mit dem Export.

### IDE-Plugin — *Tools > Optionen > "Sonar Integration"*

Neue Options-Page neben der existierenden "Static Code Analyser"-Page,
unter dem lokalisierten "Fremdhersteller" / "Third Party"-Knoten:

- Host URL, Project Key, Bearer Token (PasswordChar + Show/Hide),
  Branch, Insecure-TLS-Checkbox
- **"Detect"**-Button liest `sonar.projectKey` + `sonar.host.url`
  aus der `sonar-project.properties` des aktiven IDE-Projekts
- **"Test Connection"** läuft denselben mehrstufigen Check wie der
  CLI `--sonar-test` und rendert das Ergebnis als Monospace-Memo
- **"Open analyser.ini"** öffnet die INI im Default-Editor
- Token-Storage per DPAPI, nie im Klartext geschrieben

### *Send to Sonar* Kontext-Menüs

Zwei neue Einträge im Findings-Grid-Export-Menü:

- **"Sonar: write Generic Issue report (all findings)..."** — Bulk-
  Export von `FAll` (das vollständige Analyse-Ergebnis, nicht die
  grid-gefilterte Sicht) in einen vom User gewählten Pfad
- **"Sonar: send selected as external issue"** — Multi-Select-fähig,
  schreibt pro Finding eine JSON-Datei nach `<repo>\.sonar\external\`
  zur Abholung durch `sonar.externalIssuesReportPaths=.sonar/external/`

### Pull-Mode-Engine (UI-Binding verschoben)

[`uSonarPull`](../../StaticCodeAnalyserForm/sources/Infrastructure/uSonarPull.pas)
implementiert `GET /api/issues/search` mit einem 5-Minuten-LRU-Cache
pro `(project, file)` und einem konservativen `TSonarDedupMatcher`,
der SCA-Kinds auf bekannte SonarDelphi-Rule-Name-Fragmente mappt
(`EmptyExcept` ↔ `emptyexcept`, `UnusedUses` ↔ `unusedimport` usw).
Die Engine ist fertig; die Anbindung ans Dock-Window-Findings-Grid
(zweite Marker-Farbe, "(already in Sonar)"-Suffix) ist auf einen
späteren Release verschoben.

### PowerShell-Helfer

Zwei Scripts unter
[`StaticCodeAnalyserForm/scripts/`](../../StaticCodeAnalyserForm/scripts/):

- **`sonar-scan.ps1`** — ruft `analyser.exe --sonar-export`, deployt
  beim ersten Lauf den Rule-Catalog automatisch nach `%APPDATA%`,
  decodiert den Exit-Code in ein menschenlesbares Verdict
- **`sonar-upload.ps1`** — entschlüsselt das DPAPI-Token aus
  `analyser.ini`, ruft `sonar-scanner` mit den richtigen `-D`-Flags.
  Schalter: `-DryRun` (zeigt nur das Kommando, kein Netzwerk),
  `-DisableDelphi` (legt den SonarDelphi / Community-Fork
  DelphiSensor still, wenn er sich an Fremd-`.dproj`-Files im Scan-
  Scope verschluckt)

Implementation Notes:

- `uSonarConfig.HttpGet` nutzt `System.Net.HttpClient` mit 5 s
  Connection-Timeout / 10 s Response-Timeout damit `--sonar-test`
  beschränkt bleibt.
- `TMemIniFile` statt `TIniFile` — die INI verträgt UTF-8-BOM
  (Notepad-Default).
- `IOTAIDEThemingServices.ApplyTheme` wird in `FrameCreated`
  aufgerufen, damit die Page das aktive IDE-Theme respektiert
  (Dark / Light / Mountain Mist).

## 📚 Rule-Catalog: 22 → 59 Regeln

[`rules/sca-rules.json`](../../rules/sca-rules.json) (und sein
[Schema](../../rules/sca-rules.schema.json)) bekommen 37 neue Rule-
Einträge die jeden `TFindingKind` abdecken den die Codebase emittiert.
Highlights:

- **Alle 20 DFM-Detektoren** (`DfmDefaultName`, `DfmHardcodedCaption`,
  `DfmHardcodedDbCreds`, `DfmDuplicateBinding`, `DfmDeadEvent`,
  `DfmOrphanHandler`, `DfmEmptyBoundEvent`, `DfmSchemaMismatch`,
  `DfmCircularDataSource`, `DfmSqlFromUserInput`,
  `DfmRequiredFieldUnbound`, `DfmRequiredFieldNotVisible`,
  `DfmFieldTypeMismatch`, `DfmTabOrderConflict`, `DfmForbiddenClass`,
  `DfmDbInUiForm`, `DfmCrossFormCoupling`, `DfmLayerViolation`,
  `DfmGodHandler`, `DfmActionMismatch`)
- **Pascal-seitige neuere Detektoren** (`ConcatToFormat`,
  `WithStatement`, `ReversedForRange`, `SelfAssignment`,
  `VirtualCallInCtor`, `LengthUnderflow`, `TautologicalBoolExpr`,
  `SqlDangerousStatement`, `FormatLocaleHint`)
- **Visibility-Checks** (`CanBePrivate`, `CanBeProtected`,
  `UnusedPublicMember`, `UnusedLocalVar`, `UnusedParameter`)
- **Aggregierte DFM** (`DfmMasterDetailUnlinked`,
  `DfmDataModuleSplitHint`)
- **Custom-Rules** (`CustomRule` — Platzhalter-Eintrag für
  user-definierte Rules aus `analyser-rules.yml`)

Jeder Eintrag trägt `cleanCodeAttribute` + `impacts` (MQR-Felder)
damit SonarQube ohne manuelles Patching importiert. Das Mapping
matched die Vorschläge aus `todo-sonar.md` Phase A / P2; Rules die
dort nicht explizit aufgeführt sind, folgen derselben Logik
(`DfmHardcodedDbCreds` spiegelt `HardcodedSecret`, `DfmDeadEvent`
spiegelt `DfmSchemaMismatch`, etc).

[`docs/rules.md`](../rules.md) ist regeneriert mit der vollen 59-
Regel-Tabelle und per-Regel-Detail-Sektionen. Catalog-Schema-Version
1.1 → 1.3.

Neue Konsistenz-Tests in
[`uTestRuleCatalog`](../../StaticCodeAnalyserForm/tests/uTestRuleCatalog.pas):

- `EveryFindingKindHasRichMetadata` — jeder Kind hat non-empty
  `Name`, `ShortDescription`, `DetectorUnit` (fängt den Fall ab,
  dass `LoadFallback` einsprang)
- `JsonSeverityMatchesKindMeta` — JSON-`defaultSeverity` und
  `KIND_META.DefaultSeverity` müssen bit-gleich pro Kind sein
- `EveryFindingKindHasMqrMapping` — jeder Kind hat ein gültiges
  `cleanCodeAttribute` plus mindestens einen `impact`

## 🔧 Severity Single-Source

`TLeakFinding.SetKind(K)` in
[`uMethodd12`](../../StaticCodeAnalyserForm/sources/Common/uMethodd12.pas)
setzt Kind plus zieht Severity aus `KIND_META.DefaultSeverity`.
58 Emit-Stellen quer durch 53 Detektoren plus `uStaticAnalyzer2`
vom alten Pattern refaktoriert:

```pascal
// vorher
F.Severity := lsError;
F.Kind     := fkXxx;

// nachher
F.SetKind(fkXxx);
```

Drei Detektoren behalten ihre manuelle `F.Severity := ...` Zuweisung,
weil Severity dort kontext-abhängig ist:

- `uLeakDetector2` — confidence-bewertetes `Sev`
- `uDivByZero` — confidence-bewertetes `Sev`
- `uCustomRuleDetector` — Severity aus `analyser-rules.yml`

Zwei stille Severity-Werte als Nebeneffekt korrigiert (der Detektor-
Code widersprach dem Catalog; ab jetzt gewinnt der Catalog):

- `fkNilDeref`: Detektor emittierte `lsError`, Catalog sagte Warning
  → ab jetzt Warning. Kein existierender Test prüfte `lsError`;
  SARIF-Export nutzte schon Warning.
- `fkUnusedUses`: Detektor emittierte `lsWarning`, Catalog sagte Hint
  → ab jetzt Hint.

## 📦 Migrations-Hinweise

- **Catalog-Deployment**: kopier `rules/sca-rules.json` einmal nach
  `%APPDATA%\StaticCodeAnalyser\rules\` für pfad-unabhängiges Lookup,
  oder lass einfach `sonar-scan.ps1` einmal laufen — der deployt
  automatisch.
- **SonarDelphi / communitydelphi installiert**: `-DisableDelphi`
  an `sonar-upload.ps1` übergeben (oder die äquivalenten `-D`-Flags
  manuell setzen). Der `DelphiSensor` des Plugins crasht bei Fremd-
  `.dproj`-Files im Scan-Scope; der Schalter setzt leere Language-
  Patterns sodass der Sensor null Input bekommt und sauber endet.
- **`analyser.ini`** bekommt eine `[Sonar]`-Section plus
  `[SonarTokens]` — alte Configs ohne diese Sections funktionieren
  weiter, der Sonar-Flow fragt einfach nach Credentials.
- **Severity-Änderungen** (`fkNilDeref`, `fkUnusedUses`) können in
  CI-Gates auftauchen, die nach Severity filtern. Beide Bewegungen
  sind zum Catalog-Default hin; keine anderen Code-Pfade sind
  betroffen.
