## v0.9.1 — SonarQube Integration

Production-ready external-issues push to **SonarQube / SonarCloud**,
a major **rule-catalog expansion** (22 → 59 rules), and a
**severity single-source refactor**. Detector count grows from 41 to
**59** with 18 new Pascal / visibility / SQL / format detectors plus
all 20 DFM detectors finally landing in the documented catalog.

Three big themes:

1. **SonarQube integration** — Standalone CLI (`--sonar-export`,
   `--sonar-test`, `--sonar-init`) plus IDE plugin
   (*Tools > Options > Sonar Integration*, *Send to Sonar* context menus)
   plus PowerShell helper scripts. DPAPI-encrypted token storage. MQR
   fields (`cleanCodeAttribute` + `impacts`) populated for every rule.
2. **59 documented rules** — every emitted `TFindingKind` now has a
   full catalog entry with `cleanCodeAttribute`, `impacts`, examples,
   tags, CWE, OWASP. SARIF and Sonar Generic Issue Format reports are
   no longer missing metadata.
3. **Severity single-source** — `TLeakFinding.SetKind(K)` pulls the
   default severity from `KIND_META`. 58 detector emit sites refactored
   from the old two-line pattern. Two stale severity values fixed as a
   side effect (`fkNilDeref` Error→Warning, `fkUnusedUses`
   Warning→Hint).

## 🛰 SonarQube integration (NEW)

End-to-end pipeline for pushing SCA findings to a SonarQube server.

### `uSonarConfig` — four-source config resolver

Order of resolution (highest priority first), each source only fills
empty fields:

| Source | Standalone | IDE plugin |
|---|---|---|
| CLI flags (`--sonar-host`, `--sonar-token`, ...) | ✅ | — |
| Environment (`SONAR_HOST_URL`, `SONAR_TOKEN`, ...) | ✅ | ✅ |
| `sonar-project.properties` in the project root | ✅ | ✅ |
| `%APPDATA%\StaticCodeAnalyser\analyser.ini` `[Sonar]` + `[SonarTokens]` | ✅ | ✅ |

Token storage on Windows is DPAPI-encrypted (Current-User scope) —
only the same Windows user on the same machine can decrypt it.
Non-Windows: plaintext+Base64 fallback with a `PT:` marker, documented
in [docs/sonar-config.md](../sonar-config.md).

### `--sonar-test` health-check

Multi-stage connectivity probe:

```text
[OK]   DNS resolution: sonar.company.com -> 10.0.0.42
[OK]   HTTP /api/system/status: UP
[OK]   Token validation: valid
[OK]   Project access: my-delphi-project (visible)
Sonar connection healthy.
```

403 disambiguation: when `/api/projects/search` returns 403,
a fallback request to `/api/components/show` distinguishes
"project does not exist" (404) from "no Browse permission" (403)
and shows the right next-step in the failure message.

### `--sonar-export` Generic Issue Format writer

```powershell
analyser.exe --path . --full --base-dir . --sonar-export sca-findings.json
```

Writes a Sonar Generic Issue Format JSON with all findings. Each rule
carries:

- `engineId = "static-code-analyser"`
- `cleanCodeAttribute` (one of `LAWFUL` / `LOGICAL` / `FOCUSED` / ...
  14-value MQR taxonomy)
- `impacts: [{softwareQuality, severity}]` (MQR mode)
- `effortMinutes` derived from FindingType

Fallback path: when the catalog isn't loaded (unusual, e.g. corrupted
`rules/sca-rules.json`), the exporter emits the legacy `type` field
from `KIND_META.FindingType` instead — Sonar still accepts the JSON.

### `--sonar-init` project-template

```powershell
analyser.exe --sonar-init --path .
```

Writes a `sonar-project.properties` template with sensible defaults
and the `sonar.externalIssuesReportPaths=sca-findings.json` line ready
to consume the export.

### IDE plugin — *Tools > Options > "Sonar Integration"*

New options page next to the existing "Static Code Analyser" page,
under the localized "Fremdhersteller" / "Third Party" node:

- Host URL, Project Key, Bearer Token (PasswordChar + Show/Hide),
  Branch, Insecure-TLS toggle
- **"Detect"** button reads `sonar.projectKey` + `sonar.host.url`
  from the active IDE project's `sonar-project.properties`
- **"Test Connection"** runs the same multi-stage check as the CLI
  `--sonar-test` and renders the result in a monospace memo
- **"Open analyser.ini"** opens the INI in the default editor
- Token storage via DPAPI, never written in plaintext

### *Send to Sonar* context menus

Two new entries in the findings-grid export menu:

- **"Sonar: write Generic Issue report (all findings)..."** — bulk
  export of `FAll` (the full analyzer result, not the grid-filtered
  view) to a user-chosen path
- **"Sonar: send selected as external issue"** — multi-select-aware,
  writes per-finding JSON files into `<repo>\.sonar\external\` for
  `sonar.externalIssuesReportPaths=.sonar/external/` to pick up

### Pull-mode engine (UI binding deferred)

[`uSonarPull`](../../SCA.Engine/sources/Infrastructure/uSonarPull.pas)
implements `GET /api/issues/search` with a 5-minute LRU cache per
`(project, file)` and a conservative `TSonarDedupMatcher` that maps
SCA kinds to known SonarDelphi rule-name fragments
(`EmptyExcept` ↔ `emptyexcept`, `UnusedUses` ↔ `unusedimport`, etc).
The engine is complete; the dock-window finding-grid binding (second
marker color, "(already in Sonar)" suffix) is deferred to a later
release.

### PowerShell helpers

Two scripts under
[`StaticCodeAnalyserForm/scripts/`](../../StaticCodeAnalyserForm/scripts/):

- **`sonar-scan.ps1`** — runs `analyser.exe --sonar-export`,
  auto-deploys the rule catalog to `%APPDATA%` on first run, decodes
  the exit code into a human-readable verdict
- **`sonar-upload.ps1`** — decrypts the DPAPI token from
  `analyser.ini`, runs `sonar-scanner` with the right `-D` flags.
  Switches: `-DryRun` (prints command, no network), `-DisableDelphi`
  (silences the SonarDelphi / community-fork DelphiSensor when it
  chokes on foreign `.dproj` files in the scan scope)

Implementation notes:

- `uSonarConfig.HttpGet` uses `System.Net.HttpClient` with a 5s
  connection timeout / 10s response timeout to keep `--sonar-test`
  bounded.
- `TMemIniFile` instead of `TIniFile` so the INI tolerates UTF-8 BOM
  (Notepad default).
- `IOTAIDEThemingServices.ApplyTheme` is called from each
  `FrameCreated` so the page respects the active IDE theme
  (Dark / Light / Mountain Mist).

## 📚 Rule-catalog: 22 → 59 rules

[`rules/sca-rules.json`](../../rules/sca-rules.json) (and its
[schema](../../rules/sca-rules.schema.json)) gain 37 new rule entries
covering every `TFindingKind` the codebase emits. Highlights:

- **All 20 DFM detectors** (`DfmDefaultName`, `DfmHardcodedCaption`,
  `DfmHardcodedDbCreds`, `DfmDuplicateBinding`, `DfmDeadEvent`,
  `DfmOrphanHandler`, `DfmEmptyBoundEvent`, `DfmSchemaMismatch`,
  `DfmCircularDataSource`, `DfmSqlFromUserInput`,
  `DfmRequiredFieldUnbound`, `DfmRequiredFieldNotVisible`,
  `DfmFieldTypeMismatch`, `DfmTabOrderConflict`, `DfmForbiddenClass`,
  `DfmDbInUiForm`, `DfmCrossFormCoupling`, `DfmLayerViolation`,
  `DfmGodHandler`, `DfmActionMismatch`)
- **Pascal-side newer detectors** (`ConcatToFormat`, `WithStatement`,
  `ReversedForRange`, `SelfAssignment`, `VirtualCallInCtor`,
  `LengthUnderflow`, `TautologicalBoolExpr`, `SqlDangerousStatement`,
  `FormatLocaleHint`)
- **Visibility checks** (`CanBePrivate`, `CanBeProtected`,
  `UnusedPublicMember`, `UnusedLocalVar`, `UnusedParameter`)
- **Aggregated DFM** (`DfmMasterDetailUnlinked`,
  `DfmDataModuleSplitHint`)
- **Custom rules** (`CustomRule` — placeholder entry for user-defined
  rules loaded from `analyser-rules.yml`)

Every entry carries `cleanCodeAttribute` + `impacts` (MQR fields) so
SonarQube imports cleanly without manual patching. The mapping
matches the suggestions from `todo-sonar.md` Phase A / P2; rules not
explicitly listed there use the same logic
(`DfmHardcodedDbCreds` mirrors `HardcodedSecret`, `DfmDeadEvent`
mirrors `DfmSchemaMismatch`, etc).

[`docs/rules.md`](../rules.md) is regenerated with the full 59-rule
table and per-rule detail sections. Catalog schema version 1.1 → 1.3.

New consistency tests in
[`uTestRuleCatalog`](../../StaticCodeAnalyserForm/tests/uTestRuleCatalog.pas):

- `EveryFindingKindHasRichMetadata` — every kind has non-empty
  `Name`, `ShortDescription`, `DetectorUnit` (catches the
  `LoadFallback`-was-hit case)
- `JsonSeverityMatchesKindMeta` — JSON `defaultSeverity` and
  `KIND_META.DefaultSeverity` must be bit-equal per kind
- `EveryFindingKindHasMqrMapping` — every kind has a valid
  `cleanCodeAttribute` plus at least one `impact`

## 🔧 Severity single-source

`TLeakFinding.SetKind(K)` in
[`uMethodd12`](../../SCA.Engine/sources/Common/uMethodd12.pas)
sets Kind plus pulls Severity from `KIND_META.DefaultSeverity`.
58 emit sites refactored across 53 detectors plus `uStaticAnalyzer2`
from the old pattern:

```pascal
// before
F.Severity := lsError;
F.Kind     := fkXxx;

// after
F.SetKind(fkXxx);
```

Three detectors keep their manual `F.Severity := ...` assignment
because severity is genuinely context-dependent:

- `uLeakDetector2` — confidence-graded `Sev` variable
- `uDivByZero` — confidence-graded `Sev` variable
- `uCustomRuleDetector` — severity from `analyser-rules.yml`

Two stale severity values fixed as a side effect (the detector code
disagreed with the catalog; the catalog wins now):

- `fkNilDeref`: detector emitted `lsError`, catalog said Warning →
  now Warning. No existing test asserted `lsError`; SARIF export
  was already using Warning.
- `fkUnusedUses`: detector emitted `lsWarning`, catalog said Hint →
  now Hint.

## 📦 Migration notes

- **Catalog deployment**: drop a copy of `rules/sca-rules.json` into
  `%APPDATA%\StaticCodeAnalyser\rules\` for path-independent lookup,
  or just run `sonar-scan.ps1` once — it auto-deploys.
- **SonarDelphi / communitydelphi installed**: pass `-DisableDelphi`
  to `sonar-upload.ps1` (or set the equivalent `-D` flags by hand).
  The plugin's `DelphiSensor` crashes on foreign `.dproj` files in
  the scan scope; the switch sets empty language patterns so the
  sensor receives no input and exits cleanly.
- **`analyser.ini`** gains a `[Sonar]` section plus
  `[SonarTokens]` — old configs without these sections continue to
  work, the Sonar flow simply asks for credentials.
- **Severity changes** (`fkNilDeref`, `fkUnusedUses`) may surface in
  CI gates that filter by severity. Both moves are towards the
  catalog default; no other code paths are affected.
