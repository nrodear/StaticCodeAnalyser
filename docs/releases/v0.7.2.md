## v0.7.2 — Responsive-Layout Refactor + Detector Accuracy

Patch release on top of v0.7.1. Focus: architecture cleanup of the
plugin UI (central 3-stage layout controller instead of scattered
visibility togglers), font-scaled component heights, three
FormatMismatch detector fixes after code reviews on real
mORMot-2.4 findings, plus polish on the in-editor hover-hint overlay.

### Highlights

- **Central `TResponsiveLayoutController`** (NEW, replaces
  `TResponsiveVisibilityController`) — one class, one ClientWidth
  source (Frame.OnResize), one visibility table. Previously 5
  scattered controller instances across 4 panels with chained
  OnResize hooks; now declarative stage registration in one place:
  ```pascal
  FResp := TResponsiveLayoutController.Create(Self, Self,
             BREAKPOINT_MEDIUM, BREAKPOINT_FULL);
  FResp.RegisterCtrl(FBtnCancel,    usFull);             // FULL only
  FResp.RegisterCtrl(FLblFilter,    usMedium);           // MEDIUM and up
  FResp.RegisterCtrl(FBtnHamburger, usNarrow, usMedium); // inverse
  ```
  Has `AfterApply` callback for follow-up adjustments (sub-panel-width
  sync, SearchEdit-MinWidth sync). Result: 5 controller instances +
  chained-OnResize race conditions gone, +80 lines of code deleted,
  all visibility rules readable at a glance.

- **`TToolbarSizing` helper** (NEW) — fixes the VCL quirk that
  `TComboBox` ignores `Align.Height` (always renders at
  `ItemHeight + ~6 px`). Three static methods:
  - `HeightForFont(Font)` — derives target height from
    `Abs(Font.Height) + 11`. DPI-aware via `Font.Height` (pixels,
    no extra `ScaleW` needed). For Segoe UI 8pt = 22 px,
    9pt = 23 px, 10pt = 24 px.
  - `Apply(Ctrl, AHeight)` — sets `Constraints.MinHeight = MaxHeight`
    (VCL respects this even with active Align), and on TComboBox
    additionally `ItemHeight := AHeight - 6`. Result: buttons / edits /
    combos render uniformly.
  - `ApplyIconButton(Ctrl, AWidth, AHeight)` — additionally Width
    constraints for pixel-precise icon buttons (Browse "...",
    Hamburger ☰, Branch-Changes ⎇ all uniform 32 px).

- **Floated min-width = 500 px** (NEW) — `Frame.Constraints.MinWidth`
  + propagation to the IDE host form via `GetParentForm` in
  `FrameCreated`. Prevents pathologically narrow floats — the floated
  window can no longer shrink below the MEDIUM-stage threshold.

- **In-editor hover-hint overlay** (refined) — theme-adaptive (light/
  dark follows the active IDE theme via
  `INTACodeEditorServices.Options.BackgroundColor`), multi-line wrap
  text, severity-tinted title bar with accent stripe on the left and
  type badge on the right. Clicking a finding line marks ALL findings
  in the same file (multi-marker via
  `TDictionary<Integer, TFindingMark>`). Fully i18n via `_()`.

### Engine

- **FormatMismatch: three connected false-positive fixes** after
  code reviews on real mORMot-2.4 findings:

  1. **Bare-`%` counting** for mORMot functions (`FormatUtf8`,
     `FormatString`, `StringFormatUtf8`): these use `%` alone as
     placeholder (no type letter like `%s` / `%d`). New
     `IsBareStyle` check + second counting strategy via
     `CountPlaceholders(ABareStyle)`. Hardcoded list `BARE_STYLE_FUNCS` —
     extend there for further mORMot idiom functions.

  2. **`%%` is NOT an escape in bare style** — verified in mORMot
     source (`mormot.core.text.pas:9616 TFormatUtf8.Parse`): every
     `%` consumes one argument, `%%` = two consecutive args without
     a separator (mORMot uses this e.g. to chain WHERE-clauses).
     Standard RTL `Format` stays strict — `%%` is still an escape there.

  3. **String-literal concatenation `'a' + 'b'`** is merged before
     counting — typical for multi-line SQL strings:
     ```pascal
     FormatUtf8('SELECT ... ' + 'WHERE Id=%', [id])  // OK, 1+1 match
     ```
     Helpers `ReadStringLiteral` / `SkipSpaces`, loop until no more
     `+ '...'` follows. Non-static continuations (`+ IntToStr(x)`)
     break the merge and the detector reports only the merged prefix
     (conservative, no false positive).

  Effect on real mORMot-2.4 findings:
  | File | v0.7.1 | v0.7.2 |
  |---|---|---|
  | `dmvc-ai/.../api.impl.pas:62` | False positive (0 vs 1) | OK (1 vs 1) |
  | `dmvc-ai/.../api.impl.pas:71` | False positive (1 vs 2) | OK (2 vs 2) |
  | `dmvc-ai/.../api.impl.pas:126` | False positive (1 vs 2) | OK (2 vs 2) |
  | `mormot.orm.rest.pas:1780` | True bug, count wrong | True bug, count correct (9 vs 8) |

### IDE plugin

- **`TResponsiveVisibilityController` removed** (deprecated in v0.7.1,
  deleted in v0.7.2) — `TResponsiveLayoutController` replaces it
  completely.
- **3-stage layout** stabilised: NARROW (`<500`), MEDIUM (`500..849`),
  FULL (`>=850`). Two stage transitions instead of one — going from
  the hamburger pattern to the full UI is smoother (no abrupt jump
  to a fully populated toolbar).
- **Hamburger menu** restructured: Analyse Branch-Changes, Cancel
  Analysis, Export..., Settings..., Ignore list... (Browse "..." and
  ▶ Analyse + 📄 File stay ALWAYS visible in the toolbar, not in the
  menu). `HamburgerMenuPopup` syncs the Enabled state of Cancel +
  Branch-Changes items with the corresponding buttons — no
  double-click race while analysis runs.
- **Branch-Changes button** as icon glyph ⎇ (32 px) instead of a
  caption button (104 px). The "Branch-Changes" caption + full text
  remain available in the tooltip and the hamburger menu.
- **`TAnalyserDockableForm.FrameCreated`** propagates the frame's
  MinWidth/MinHeight to the host form via `GetParentForm` — the
  IDE floating container now respects the minimum width on resize.
- **Tile row**: all 9 tiles uniform 55 px wide (was 65/72/72 per
  Severity / Type / Detector tile previously). Consistent Sonar-style
  row. Stage registration: 4 essentials always, +5 from MEDIUM
  upwards (Read errors, Bugs, Security, Duplicates, Cyclomatic).
- **Toolbar padding** reduced (`TB_PADDING_TB` 2→1) — toolbar rows
  take 2 px less vertical space each.

### Known limitations

- **`%%` heuristic in bare style** is conservative: every `%` counts
  regardless of context. With very exotic mORMot patterns (e.g.
  `FormatUtf8('%' + Var, [x])`) the detector may be off if `Var`
  contains a `%` at compile time — but this is a) bare-style consistent
  (mORMot would also misinterpret it) and b) hard to detect statically.
- **Bare-style function list** is hardcoded to
  `formatutf8` / `formatstring` / `stringformatutf8`. If further mORMot
  helpers (`StringFormatBuffer`, `FormatShort`, `FormatToShort`)
  appear, add them to `BARE_STYLE_FUNCS`. Could later be made
  configurable via `[Detectors] BareFormatFunctions=...`.

  FormatFunctions` from v0.7.1 still works — bare-style detection
  is orthogonal via the function name.
- **HTML / JSON / CSV exports** unchanged.

  severity configuration** remain valid.
