# Internationalization (i18n)

The plugin uses **dxgettext** (a Delphi port of GNU gettext) for UI
translation. Translations live as `.po` files under `i18n/` and can be
edited with [poEdit](https://poedit.net/) or any compatible tool.

## Architecture

```
StaticCodeAnalyserForm/sources/uLocalization.pas    Thin wrapper around dxgettext
                                                    Compiles with OR without
                                                    dxgettext installed.

i18n/template.pot                                   Master template (English source)
i18n/de.po                                          German translation
i18n/en.po                                          English baseline (mostly identity)
i18n/fr.po                                          French translation

locale/<lang>/LC_MESSAGES/default.mo                Compiled binary (generated by msgfmt)
                                                    -- only this is loaded at runtime --
```

The wrapper exposes three things:

```pascal
uses uLocalization;

_('Start analysis')                       // returns translated string
_('%d findings', [Count])                 // translate then format
SetLanguage('de')                         // switch locale at runtime
```

`uLocalization` is **safe to use immediately** ‚Äî without dxgettext
installed it returns the source string unchanged. Adding `_('...')`
wrappers to source files breaks nothing.

## Activation steps

Once you actually want to ship localized builds:

### 1. Install dxgettext

Clone or download from <https://github.com/sjrd/dxgettext> and add the
`dxgettext/Source/` folder to your IDE's library path or to the plugin's
`DCC_UnitSearchPath`.

The minimum needed for runtime is the unit `gnugettext.pas`. Tooling
binaries (`ggdxgettext.exe`, `msgfmt.exe`) come bundled in the
`dxgettext/Bin/` folder of the same repository.

### 2. Activate the wrapper

Edit `StaticCodeAnalyserIDE.dpk` (or set in **Project Options ‚Üí
Conditional Defines**):

```
{$DEFINE USE_GETTEXT}
```

After this, `uLocalization._()` routes through `gnugettext.dgettext`
instead of being a passthrough.

### 3. Set the language at frame creation

Somewhere in `TAnalyserDockableForm.FrameCreated` (or your preferred
entry point):

```pascal
uses uLocalization;

procedure TAnalyserDockableForm.FrameCreated(AFrame: TCustomFrame);
begin
  // Read user choice from %APPDATA%\StaticCodeAnalyser\app.ini, env var,
  // or whatever you prefer. 'de'/'en'/'fr'/'' (= system default).
  SetLanguage(ReadLanguageFromIni);
  ...
end;
```

### 4. Generate the .po template from source

In a shell at the repo root (with `dxgettext/Bin` on PATH):

```bash
ggdxgettext -d default ^
  StaticCodeAnalyserForm\sources\*.pas ^
  StaticCodeAnalyserIDE\*.pas ^
  -o i18n\template.pot
```

This scans every `_('‚Ä¶')` in the listed source files and writes the
extracted strings to `template.pot`.

### 5. Merge new strings into existing locale files

```bash
msgmerge -U i18n\de.po i18n\template.pot
msgmerge -U i18n\fr.po i18n\template.pot
```

`-U` updates in-place: keeps existing translations, adds new
`msgid` entries marked as untranslated, comments out obsolete ones.

### 6. Translate

Open `i18n/de.po` (or `fr.po`) in poEdit. Walk through untranslated
items, save. Or edit the file directly ‚Äî the format is plain text:

```po
msgid "Start analysis"
msgstr "Analyse starten"
```

### 7. Compile to binary `.mo`

dxgettext's runtime loads `.mo` files (binary, compact). Compile every
`.po` file:

```bash
msgfmt i18n\de.po -o locale\de\LC_MESSAGES\default.mo
msgfmt i18n\fr.po -o locale\fr\LC_MESSAGES\default.mo
```

The `locale/<lang>/LC_MESSAGES/default.mo` directory layout is gettext
convention. dxgettext expects exactly this path next to the EXE/DLL.

### 8. Ship the locale folder with the plugin

For the IDE plugin (`.bpl`), place the `locale/` folder next to
`$(BDSCOMMONDIR)\Bpl\StaticCodeAnalyserIDE.bpl`. For the standalone
EXE, place it next to `analyser.d12.exe`.

## Day-to-day workflow

1. **Add a new UI string in code**:
   ```pascal
   Btn.Caption := _('My new button');
   ```
2. **Regenerate `template.pot`** (step 4 above)
3. **Merge into locale files** (step 5)
4. **Translate** the new entry in poEdit
5. **Compile** to `.mo` (step 7)
6. **Test** with `SetLanguage('de')` etc.

## Rules of the road

### Keep source strings English

The `msgid` is always the source string. Always English in this project
‚Äî even when the developer is German-speaking. This way the source code
is internationally readable and the translation files have a stable
canonical form.

### Don't put logic on translated strings

```pascal
// WRONG ‚Äî logic depending on translated text
if Severity = _('Error') then ...

// RIGHT ‚Äî logic on enum, only display goes through _()
if SeverityFromText(Severity) = fsError then ...
LblFoo.Caption := _('Error');
```

The `TFindingSeverity` enum and `SeverityFromText`/`SeverityToText`
helpers in `uAnalyserTypes.pas` already enforce this for severity
strings. Apply the same pattern for any other piece of "data" that
happens to be displayed as text.

### Keep format placeholders identical across translations

```po
msgid "%d / %d findings"
msgstr "%d / %d Befunde"     # OK ‚Äî same %d count, same order

# WRONG ‚Äî translator dropped a %d
# msgstr "%d Befunde"
```

Verify by running poEdit's built-in validation (it warns on this).

### Don't concatenate translated fragments

```pascal
// WRONG ‚Äî German word order requires different phrasing
s := _('File ') + Name + _(' has been modified');

// RIGHT ‚Äî full sentence with placeholder
s := Format(_('File %s has been modified'), [Name]);
```

### One file, one purpose

Source code stays in one source language (English). Don't put
`{$IFDEF GERMAN}` around code blocks. Don't ship per-language EXE
builds. The runtime decides the language.

## Known limitations / things still to do

- **Detector finding messages** are not yet wrapped in `_()`. There are
  ~40 messages across the detector units (`uMagicNumbers.pas`,
  `uLongMethod.pas`, etc.) that need instrumenting before German /
  French users see localized findings. Tedious but mechanical.
- **`uFixHint.pas`** before/after texts are also unwrapped. Same
  treatment needed.
- **`.dfm`-defined captions** (the standalone Form's `Caption`) are not
  picked up by `_()` ‚Äî dxgettext has a separate
  `TranslateComponent(Self)` mechanism that needs to be invoked in the
  Form's `OnCreate`. See the dxgettext docs.
- **Plural forms**: gettext supports them via `_n('1 finding', '%d findings', N)`.
  Not yet used here; trivial to adopt where needed.

## Why this approach

Compared to the alternatives:

| Approach | Why not |
|----------|---------|
| Built-in `resourcestring` + Delphi Translation Manager | TLM workflow is clunky; not Git-friendly; switching language at runtime is awkward |
| Custom INI-based lookup | Reinventing gettext; no plural support, no validation, no tooling |
| Hardcoded multi-language (`{$IFDEF DE}`) | Doesn't scale; doesn't support runtime switching; unmaintainable beyond two languages |

dxgettext gives us:

- Industry-standard format (`.po`)
- poEdit/Crowdin/Lokalise support out of the box
- Plural forms, format validation, fuzzy matching
- Live language switching (`SetLanguage('fr')` at runtime)
- Zero overhead when the wrapper is a passthrough ‚Äî adoption is incremental
